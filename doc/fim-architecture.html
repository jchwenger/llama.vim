Overview

llama.vim provides real-time code completions using a local llama.cpp server with FIM (Fill-in-the-Middle) models. Three pillars:

🔥 Local Context

Extracts n_prefix lines before and n_suffix lines after the cursor, forming the prefix/middle/suffix structure FIM models consume.

💾 LRU Cache

SHA-256 keyed cache (250 entries) with fuzzy matching over the last 128 typed characters — instant recall without server round-trips.

🔄 Ring Buffer

Sliding window of code chunks gathered from yanks, buffer transitions, and file saves. Provides cross-file global context with similarity-based eviction.

⚡ Speculative Warm-up

After a cached hint renders, a background request warms the server KV cache for the next cursor position, making subsequent completions feel instant.

Request Flow

Every cursor movement in Insert mode triggers this pipeline:

CursorMovedI → s:on_move()
llama#fim_hide() — clear previous suggestion
s:fim_try_hint(pos_x, pos_y)
Check LRU cache (exact hash + fuzzy 128-char)
✓ Cache hit
s:fim_render()
llama#fim(prev) — speculative warm-up
✗ Cache miss
llama#fim() — build context + ring buffer
Async curl → llama.cpp /infill
s:fim_on_response() → cache insert
s:fim_try_hint() → s:fim_render()
1. CursorMoveI→ 2. fim_hide()→ 3. fim_try_hint()→ 4. llama#fim()→ 5. curl /infill→ 6. cache insert→ 7. fim_render()

Trigger Mechanism

Two autocmds fire on every CursorMovedI:

autocmd CursorMovedI * call s:on_move()                       " try cache first
autocmd CursorMovedI * call llama#fim(-1,-1,v:true,[],v:true) " fire if needed

The first clears the hint and checks cache. The second fires an async request if cache misses. This dual-trigger ensures instant display from cache while the server fetches fresh completion in parallel.

Max line suffix

If more than max_line_suffix (default 8) chars remain on the line, auto-trigger is suppressed — user is likely mid-word.

Request debounce

If s:current_job_fim is running, the request is debounced with a 100ms retry timer to prevent request storms.

LRU Cache System

g:cache_data     = {}      " hash → raw JSON
g:cache_lru_order = []     " ordered keys (oldest first)

Key = sha256(prefix + 'Î' + middle + 'Î' + suffix). The Î matches the FIM token used by models.

Multi-hash generation

Each request computes 4 hashes — full context plus 3 variants with 1–3 prefix lines stripped. Handles the case where the user scrolled down from the original position:

" Hash 0: sha256(prefix + middle + 'Î' + suffix)
" Hash 1: sha256(prefix[1:] + middle + 'Î' + suffix)
" Hash 2: sha256(prefix[2:] + middle + 'Î' + suffix)
" Hash 3: sha256(prefix[3:] + middle + 'Î' + suffix)

On eviction (>250 entries), the least recently used key is removed. Every cache_get moves the key to the end of the LRU list.

Fuzzy Matching

When no exact hash match, s:fim_try_hint() walks back up to 128 characters and checks if a cached completion exists whose content matches what the user has already typed:

Scenario: cached = function fetchData(url) {, user typed function fetch

function fetchData(url) { cached completion
Buffer: function fetch❯ cursor here
Ghost: Data(url) { remainder shown at cursor

Interactive demo: type to match the cached completion

return items.map(i => i.price) cached completion
Buffer: ❯ cursor
Ghost: remainder
Type to see the match 
for i in range(128)
  l:removed = l:pm[-(1+i):]            " last (i+1) chars typed
  l:ctx_new = l:pm[:-(2+i)] + 'Î' + l:suffix
  l:cached  = s:cache_get(sha256(l:ctx_new))
  if cached.content[0:i] ==# l:removed " typed matches cached start
    l:remainder = cached.content[i+1:]  " show the rest as ghost text
Key insight: As you type characters that match a previously cached completion, the plugin shows the remaining portion as ghost text — all without hitting the server. The fuzzy_offset is stored so cycling completions applies the same trimming.

Similarity eviction

Chunks compared using Dice coefficient over whitespace-split tokens:

similarity = 2 × |common tokens| / (|tokens₀| + |tokens₁|)

" Thresholds:
0.9  → evict from queued + active (on pick_chunk)
0.5  → evict from active only     (during FIM request)
Why 0.5 for FIM? Chunks similar to current context (even at lower similarity) distort completions by making the LLM repeat on-screen code.

Speculative Pipeline

After a cached hint renders, a background speculative request fires:

Cache hit → s:fim_render() → ghost text shown
Background: speculative warm-up
llama#fim(pos, prev_completion, use_cache=true)
s:fim_ctx_local — rebuild context as if completion typed
Server warms KV cache for next position
Response cached → instant on next cursor move

The speculative request passes a:prev (the displayed completion) to s:fim_ctx_local(), which rebuilds context as if the completion was already inserted. The server caches this context, so accepting the completion and moving on feels instant.

Two-phase timing: First request = 250ms (fast). Speculative follow-up = 1000ms (thorough).

Accept Modes

full Tab

Entire multi-line suggestion. Replaces line suffix, appends remaining lines. Cursor to end of last line.

line S-Tab

First line only. Cursor to end. If multi-line, simulates Enter for next line.

word leader+ll]

First word (whitespace + non-whitespace) from suggestion. Cursor to end of word.

Cycling

When n_cmpl > 1, cycle with <C-J> (next) / <C-K> (prev). Uses inoremap <expr> to stay in Insert mode (<C-O> would trigger InsertLeavePre → fim_hide).

Timing & Throttling

Debounce

Running job → 100ms retry. Prevents request storms.

Two-phase

First: 250ms predict. Follow-up: 1000ms.

Ring update

Every 1000ms. Skipped if insert mode + cursor moved <3s ago.

Chunk pick

Only when cursor moved >32 lines from last pick.
