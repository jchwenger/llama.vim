Overview

The debug subsystem provides a timestamped, foldable log pane that tracks key events in the plugin lifecycle — initialization, FIM completions, instruction requests, and errors.

Logs are collected in-memory and flushed to a scratch buffer on a deferred timer to avoid UI thrashing from rapid log writes.

Plugin event fires
call llama#debug_log(msg, details)
llama_debug#log(msg, details)
Build log block → prepend to s:debug.log
s:flush_sched() — 50 ms deferred timer
s:flush() — write to scratch buffer
Two modules — autoload/llama_debug.vim (backend) and autoload/llama.vim (thin wrappers).

Log Entry Flow

1. Entry creation

When llama_debug#log(msg, details) is called:

Normalize details

If the optional second argument is not a list, it is converted via split(string(details), "\n")

Build header

Format: HH:MM:SS | msg. If details exist, first detail line is appended: HH:MM:SS | msg | details[0]

Build block

With details → foldable block: header {{{, detail lines, }}}. Without details → single line.

Prepend to s:debug.log

Blocks are inserted at index 0 (newest first). Capped at s:debug.max_lines (1024) entries.

Schedule flush

Set s:debug.dirty = 1 and call s:flush_sched() if no flush is already pending.

2. Log block structure

Each log block is a list of strings:

[llama.vim-debug]
14:23:01 | fim_on_response | int main() { {{{
  return 0;
  }
}}}

14:22:58 | plugin enabled

14:22:57 | llama.vim initializing ...

Deferred Flush

Rapid successive llama#debug_log() calls (e.g., from streamed instruction responses) would thrash the UI if each write triggered an immediate buffer update. The deferred flush pattern coalesces these:

llama_debug#log() called × N
s:debug.dirty = 1, s:flush_sched()
First call: timer_start(50ms, → s:flush()).
Subsequent calls: s:debug.flush != -1 → no-op
50 ms elapsed → s:flush()
Flatten all blocks → setbufline(debug_bufnr, 1, flat_lines)
Reset dirty = 0, flush = -1
Key insight: Multiple log calls within the 50ms window share a single buffer write. The s:debug.flush timer ID is the guard — if it's -1, a new timer is created; otherwise the call returns immediately. 
" flush_sched — coalesces rapid writes
function s:flush_sched() abort
    if s:debug.flush != -1
        return          " timer already pending → skip
    endif
    let s:debug.flush = timer_start(50, {-> s:flush()}, {'repeat': 0})
endfunction

State object

let s:debug = {
    \ 'bufnr':     -1,       " buffer number (-1 = not created)
    \ 'log':       [],       " list of log blocks (each block = list of strings)
    \ 'max_lines': 1024,     " cap on number of log blocks
    \ 'flush':     -1,       " timer ID (-1 = no flush pending)
    \ 'dirty':      1,       " flag: log changed since last flush
    \ }

Buffer lifecycle

bufnr = -1 → s:ensure_buf() → botright new → s write to → setbufline() → user closes → bufhidden=hide → :LlamaDebugToggle → botright sbuffer

Plugin Wrappers

The main autoload/llama.vim module delegates all debug operations to the dedicated backend:

" autoload/llama.vim — thin delegation layer

function! llama#debug_log(msg, ...) abort
    return call('llama_debug#log', [a:msg] + a:000)
endfunction

function! llama#debug_toggle() abort
    return llama_debug#toggle()
endfunction

function! llama#debug_clear() abort
    return llama_debug#clear()
endfunction

function! llama#debug_setup() abort
    return llama_debug#setup()
endfunction
This separation keeps the debug logic isolated. The main plugin code simply calls llama#debug_log() wherever diagnostics are needed, without knowing about buffers, timers, or folds.

Design Decisions

1 In-memory log

All log blocks are kept in s:debug.log as a list of lists. The buffer is only a rendering target, not the source of truth. This allows efficient prepend operations and easy clearing.

2 Deferred flush

A 50ms timer coalesces rapid writes. Multiple log() calls within the window share one setbufline() call, avoiding buffer thrashing.

3 Foldable blocks

Multi-line details use {{{/}}} fold markers. Folds start closed (foldlevel=0), keeping the log compact while preserving detail on demand.

4 Newest first

Log blocks are insert()-ed at index 0. The most recent entries appear at the top of the buffer — where the user's eyes naturally start.

5 Ring buffer cap

s:debug.max_lines (1024) caps the in-memory log to prevent unbounded growth during long sessions.

6 Module separation

Debug logic is isolated in llama_debug.vim. The main plugin delegates through thin wrappers, keeping concerns cleanly separated.
